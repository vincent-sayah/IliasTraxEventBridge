# V0.8 lot 1 — journal de diagnostic des refus

## Objectif

La V0.7 filtre correctement les traces xAPI avant insertion dans l'outbox :

```text
statement xAPI autorisé = cours activé ET ressource activée
```

La conséquence fonctionnelle est correcte, mais l'exploitation ne voit pas toujours pourquoi une trace n'a pas été générée.

Le lot 1 de la V0.8 ajoute donc un journal de diagnostic des refus.

## Nouvelle table

Table courte compatible avec les contraintes de nommage ILIAS :

```text
evnt_evhk_itxeb_dlog
```

Champs principaux :

| Champ | Rôle |
|---|---|
| `id` | Identifiant technique du refus |
| `created_at`, `created_ts` | Date de l'événement refusé |
| `reason` | Motif métier du refus |
| `event_type` | Type logique de trace attendue |
| `component`, `event_name` | Source ILIAS |
| `user_id` | Utilisateur concerné |
| `course_ref_id`, `course_obj_id` | Cours résolu |
| `ref_id`, `obj_id`, `obj_type` | Ressource concernée |
| `source_table`, `source_id` | Origine technique du signal |
| `payload_json` | Contexte utile au diagnostic |
| `request_uri`, `http_method` | Contexte HTTP si disponible |

## Motifs de refus

Motifs initialement journalisés :

| Motif | Signification |
|---|---|
| `not_in_course` | L'événement ne concerne pas un objet contenu dans un cours |
| `missing_course_context` | Le cours n'a pas pu être résolu |
| `missing_resource_context` | La ressource n'a pas pu être résolue |
| `course_not_configured` | Le cours n'a pas encore de configuration xAPI explicite |
| `course_disabled` | Le cours est configuré mais désactivé |
| `resource_not_configured` | La ressource n'a pas encore de configuration explicite |
| `resource_disabled` | La ressource est configurée mais désactivée |
| `unsupported_object_type` | Le signal est dans un cours mais ne permet pas de produire un statement supporté |

## Sources couvertes

### EventHook

Les refus sont journalisés depuis `ilIliasTraxEventBridgeEventRouter` lorsque :

- l'événement n'est pas dans un cours ;
- le cours ou la ressource est refusé par la configuration V0.7 ;
- le statement ne peut pas être créé.

Origine technique :

```text
source_table = evnt_evhk_itxeb_log
source_id    = event_log_id
```

### read_event

Les refus sont journalisés depuis `ilIliasTraxEventBridgeReadEventTracker` lorsque :

- la consultation n'est pas résolue dans un cours ;
- le cours ou la ressource est refusé par la configuration V0.7 ;
- le statement ne peut pas être créé.

Origine technique :

```text
source_table = read_event
source_id    = obj_id
```

## Non-régression attendue

Le journal des refus ne doit jamais bloquer :

- la navigation ILIAS ;
- la génération des statements autorisés ;
- l'insertion outbox ;
- le cron ILIAS.

Si la table `evnt_evhk_itxeb_dlog` n'existe pas encore, le repository retourne simplement `0` et ne bloque rien.

## Installation / mise à jour

Après mise à jour du code, exécuter la mise à jour du plugin depuis l'administration ILIAS afin d'appliquer l'étape SQL `<#6>`.

Contrôle SQL :

```sql
SHOW TABLES LIKE 'evnt_evhk_itxeb_dlog';
```

ou :

```sql
DESCRIBE evnt_evhk_itxeb_dlog;
```

## Validation fonctionnelle proposée

Préparer un cours avec :

- cours activé ;
- une ressource activée ;
- une ressource désactivée.

Puis ouvrir une ressource désactivée et lancer le cron si nécessaire.

Contrôle :

```sql
SELECT id, created_at, reason, event_type, user_id, course_ref_id, ref_id, obj_id, obj_type, source_table, source_id
FROM evnt_evhk_itxeb_dlog
ORDER BY id DESC
LIMIT 30;
```

Résultat attendu pour une ressource désactivée :

```text
reason = resource_disabled
```

Contrôle de synthèse :

```sql
SELECT reason, COUNT(*) AS total
FROM evnt_evhk_itxeb_dlog
GROUP BY reason
ORDER BY total DESC, reason ASC;
```
