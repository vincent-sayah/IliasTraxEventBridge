# IliasTraxEventBridge — plan de développement V0.6

## Point de départ

La branche `v0.6` démarre depuis la base stable `v0.5.5`.

La V0.5.5 a stabilisé :

- la journalisation brute des EventHooks ILIAS dans `evnt_evhk_itxeb_log` ;
- l'outbox xAPI locale `evnt_evhk_itxeb_out` ;
- le cron `itxeb_send_outbox_to_trax` ;
- le filtrage métier « objet contenu dans un cours uniquement » ;
- le tracking d'exploitation réelle via `read_event` ;
- l'anti-doublon `evnt_evhk_itxeb_read` ;
- l'exclusion des traces parasites `root` / `crs` issues de `Tracking:updateStatus` génériques.

## Objectif V0.6

La V0.6 doit rendre les statements xAPI plus exploitables dans TRAX, sans casser le socle stable V0.5.5.

Objectifs principaux :

1. enrichir les statements avec les titres ILIAS utiles ;
2. ajouter des URL ILIAS consultables ;
3. mieux exploiter les métriques `read_event` ;
4. affiner les verbes xAPI et les activity types ;
5. préparer les diagnostics, filtres et purges configurables.

## Lot 1 — enrichissement xAPI objet et cours

### État

Réalisé sur la branche `v0.6`.

### Données ajoutées

- `object_title` : titre de l'objet consulté ou tracé ;
- `course_title` : titre du cours parent ;
- `object_url` : URL ILIAS de l'objet quand un `ref_id` est disponible ;
- `course_url` : URL ILIAS du cours parent quand `course_ref_id` est disponible.

### Emplacement dans le statement

- `object.definition.name` : titre objet en priorité, fallback lisible si absent ;
- `context.contextActivities.parent` : activité cours parent ;
- `context.extensions` : identifiants, titres et URL techniques ;
- `result.extensions` : métriques `read_event`.

## Lot 2 — exploitation renforcée de `read_event`

### État

Réalisé partiellement sur la branche `v0.6`.

### Champs valorisés

- `read_count` ;
- `spent_seconds` ;
- `first_access` ;
- `last_access` / `read_event_last_access`.

### Ajouts V0.6

- `read_event_first_access` est disponible dans `result.extensions` et `context.extensions`.
- `spent_seconds` est conservé dans les extensions et alimente aussi `result.duration` au format ISO 8601 lorsque la valeur est supérieure à zéro.

### Points d'attention

- `spent_seconds` peut rester à `0` selon les objets ILIAS ;
- `read_count` est utile pour distinguer première consultation et reconsultation ;
- `first_access` sert à l'analyse, sans changer à lui seul le verbe xAPI.

## Lot 3 — wording xAPI et familles de statements

### État

Réalisé partiellement sur la branche `v0.6`.

### Familles ajoutées dans les extensions xAPI

- `statement_family` : famille analytique du statement ;
- `interaction_type` : type d'interaction fonctionnelle (`read`, `visit`, `view`, `interact`, `download`, `assessment_progress`, etc.) ;
- `repository_object_family` : famille d'objet ILIAS pour les traces issues de `read_event`.

### Mapping actuel

| Objet ILIAS | Verbe xAPI V0.6 | `interaction_type` | `statement_family` |
|---|---|---|---|
| `blog` | `read` / `a lu le blog` | `read` | `repository_blog_access` |
| `wiki` | `read` / `a lu le wiki` | `read` | `repository_wiki_access` |
| `htlm` | `read` / `a lu le module HTML` | `read` | `repository_html_module_access` |
| `lm` | `read` / `a lu le module` | `read` | `repository_learning_module_access` |
| `sahs` | `launched` / `a lancé le module SCORM` | `launch` | `repository_scorm_module_access` |
| `webr` | `visited` / `a visité le lien web` | `visit` | `repository_web_link_access` |
| `mcst` | `viewed` / `a visionné le médiacast` | `view` | `repository_media_access` |
| `frm` | `interacted` / `a interagi avec le forum` | `interact` | `repository_forum_access` |
| `file` | `downloaded` / `a téléchargé` | `download` | `file_download` |
| `tst` | `attempted`, `passed`, `failed` | `assessment_progress` | `test_tracking` |

### Compatibilité outbox

La colonne SQL `event_type` reste compatible avec les validations V0.5/V0.6 :

- `repository_object_access` pour les consultations issues de `read_event` ;
- `file_downloaded` pour les téléchargements ;
- `test_tracking_status` pour les traces de test.

La séparation fine est portée dans le JSON xAPI afin d'améliorer l'analyse TRAX sans casser les requêtes SQL existantes.

## Lot 4 — extensions techniques et diagnostic outbox

### État

Réalisé partiellement sur la branche `v0.6`.

### Extensions ajoutées

L'outbox enrichit maintenant `context.extensions` juste avant l'insertion SQL, car c'est à ce moment que l'identifiant local `outbox_id` est connu.

Extensions ajoutées :

- `outbox_id` : identifiant local dans `evnt_evhk_itxeb_out` ;
- `outbox_table` : nom de la table outbox locale ;
- `event_log_id` : identifiant du journal brut EventHook, ou `0` pour les traces synthétiques `read_event` ;
- `statement_uuid` : copie du UUID xAPI pour recherche facilitée dans TRAX ;
- `event_record_source` : source logique (`read_event_tracker`, `event_hook_log` ou `synthetic`) ;
- `source_table` : table ILIAS/plugin source (`read_event` ou `evnt_evhk_itxeb_log`) ;
- `deduplication_key` : clé explicite de déduplication ou de rapprochement.

### Stratégie de déduplication documentée

- Pour `read_event` : `read_event:{obj_id}:{user_id}:{read_event_last_access}:{read_count}`.
- Pour les EventHooks journalisés : `event_log:{event_log_id}`.
- Pour les autres cas synthétiques : clé construite à partir du composant, de l'événement, de l'utilisateur, du ref_id, de l'obj_id, du type objet et du timestamp.

### Compatibilité

Aucune colonne SQL n'est ajoutée. Les extensions sont uniquement portées dans le JSON xAPI pour faciliter les recherches TRAX et le diagnostic croisé avec l'outbox locale.

### Vérifications JSON xAPI

Sur un statement récent, vérifier dans `statement_json` :

- `context.extensions.statement_family` ;
- `context.extensions.interaction_type` ;
- `context.extensions.repository_object_family` pour les objets `read_event` ;
- `result.duration` lorsque `spent_seconds > 0` ;
- `verb.display.fr-FR` plus précis que le libellé générique `a consulté` ;
- `context.extensions.outbox_id` ;
- `context.extensions.event_record_source` ;
- `context.extensions.source_table` ;
- `context.extensions.deduplication_key`.

### Vérifications fonctionnelles

- Consultation blog dans un cours : statement enrichi avec titre objet et cours ;
- Consultation wiki dans un cours : statement enrichi ;
- Consultation lien web dans un cours : statement enrichi ;
- Consultation mediacast dans un cours : statement enrichi ;
- Téléchargement fichier dans un cours : statement enrichi ;
- Test commencé/réussi/échoué : statement enrichi ;
- Objet hors cours : aucune ligne outbox ;
- `root` / `crs` parasites : aucune nouvelle ligne.

## Règle de stabilité

La V0.6 doit être développée sur la branche `v0.6`. La branche `main` ne doit être avancée qu'après validation complète et tag stable.
