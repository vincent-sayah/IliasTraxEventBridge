# IliasTraxEventBridge — plan de développement V0.6

## Point de départ

La branche `v0.6` démarre depuis la base stable `v0.5.5`.

La V0.5.5 a stabilisé :

- la journalisation brute des EventHooks ILIAS dans `evnt_evhk_itxeb_log` ;
- l'outbox xAPI locale `evnt_evhk_itxeb_out` ;
- le cron `itxeb_send_outbox_to_trax` ;
- le filtrage métier « objet contenu dans un cours uniquement » ;
- le tracking d'exploitation réelle via `read_event` ;
- l'anti-doublon `evnt_evhk_itxeb_read` ;
- l'exclusion des traces parasites `root` / `crs` issues de `Tracking:updateStatus` génériques.

## Objectif V0.6

La V0.6 doit rendre les statements xAPI plus exploitables dans TRAX, sans casser le socle stable V0.5.5.

Objectifs principaux :

1. enrichir les statements avec les titres ILIAS utiles ;
2. ajouter des URL ILIAS consultables ;
3. mieux exploiter les métriques `read_event` ;
4. affiner les verbes xAPI et les activity types ;
5. préparer les diagnostics, filtres et purges configurables.

## Lot 1 — enrichissement xAPI objet et cours

### Données à ajouter

- `object_title` : titre de l'objet consulté ou tracé ;
- `course_title` : titre du cours parent ;
- `object_url` : URL ILIAS de l'objet quand un `ref_id` est disponible ;
- `course_url` : URL ILIAS du cours parent quand `course_ref_id` est disponible.

### Emplacement cible dans le statement

- `object.definition.name` : titre objet en priorité, fallback lisible si absent ;
- `context.contextActivities.parent` : activité cours parent ;
- `context.extensions` : identifiants, titres et URL techniques ;
- `result.extensions` : métriques `read_event`.

## Lot 2 — exploitation renforcée de `read_event`

Champs à valoriser :

- `read_count` ;
- `spent_seconds` ;
- `first_access` ;
- `last_access` / `read_event_last_access`.

Points d'attention :

- `spent_seconds` peut rester à `0` selon les objets ILIAS ;
- `read_count` est utile pour distinguer première consultation et reconsultation ;
- `first_access` peut servir à générer une extension d'analyse, sans forcément changer le verbe xAPI.

## Lot 3 — wording xAPI et familles de statements

Familles proposées :

| Famille | Source | `event_type` pressenti | Verbe xAPI pressenti |
|---|---|---|---|
| Consultation objet dépôt | `read_event` | `repository_object_access` | `experienced` |
| Téléchargement fichier | EventHook `sendfile` | `file_downloaded` | `downloaded` si retenu, sinon `experienced` |
| Test commencé | `Tracking:updateStatus` | `test_tracking_status` | `attempted` |
| Test réussi | `Tracking:updateStatus` | `test_tracking_status` | `passed` |
| Test échoué | `Tracking:updateStatus` | `test_tracking_status` | `failed` |
| Progression module | à étudier | à définir | `progressed` / `completed` |

## Lot 4 — configuration et exploitation

Évolutions à étudier après le lot xAPI :

- filtres globaux par type d'objet ;
- activation/désactivation par cours ;
- page de diagnostic TRAX ;
- purge configurable des anciens événements/outbox ;
- indicateurs de volume : outbox générée, envoyée, échouée, `read_event` traité.

### Vérifications fonctionnelles

- Consultation blog dans un cours : statement enrichi avec titre objet et cours ;
- Consultation wiki dans un cours : statement enrichi ;
- Consultation lien web dans un cours : statement enrichi ;
- Consultation mediacast dans un cours : statement enrichi ;
- Téléchargement fichier dans un cours : statement enrichi ;
- Test commencé/réussi/échoué : statement enrichi ;
- Objet hors cours : aucune ligne outbox ;
- `root` / `crs` parasites : aucune nouvelle ligne.

## Règle de stabilité
