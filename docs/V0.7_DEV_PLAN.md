# Plan de développement — IliasTraxEventBridge v0.7

## Objectif principal

La V0.7 introduit le pilotage fonctionnel des traces xAPI au niveau des cours ILIAS.

Un administrateur ou responsable de cours doit pouvoir :

- activer ou désactiver les traces xAPI pour un cours ;
- choisir quelles ressources du cours sont tracées ;
- éviter la génération de statements pour les cours ou ressources non autorisés ;
- garder une administration globale minimale du plugin pour les paramètres TRAX et la supervision.

La V0.7 transforme donc le plugin d'une collecte globale vers une collecte pilotée par cours.

## Périmètre fonctionnel

### Inclus en V0.7

- Configuration xAPI par cours.
- Configuration xAPI par ressource dans un cours.
- Écran ou onglet de configuration dans le contexte de l'objet cours.
- Filtrage avant insertion dans l'outbox.
- Traçabilité de la décision de filtrage pour faciliter le diagnostic.
- Documentation de validation dédiée V0.7.

### Hors périmètre V0.7

- Retry manuel des `failed` depuis l'admin.
- Purge configurable automatique.
- Page détail complète d'un statement outbox.
- Refonte du modèle analytique xAPI.

Ces sujets sont candidats pour V0.8 ou V0.9.

## Emplacement UI attendu

L'interface doit être accessible depuis le cours, idéalement dans :

```text
Objet cours > Paramètres > section TRAX / xAPI
```

ou, si l'API ILIAS 10 ne permet pas une insertion propre dans les paramètres natifs :

```text
Objet cours > onglet ou sous-écran TRAX / xAPI
```

L'objectif fonctionnel reste identique : la configuration doit appartenir au cours, pas seulement à l'administration globale du plugin.

## Permissions attendues

Seuls les utilisateurs autorisés à administrer le cours doivent pouvoir modifier la configuration xAPI.

À vérifier dans ILIAS 10 :

- administrateur global ;
- administrateur du cours ;
- propriétaire ou responsable disposant des droits de modification du cours.

Un utilisateur sans droit d'administration du cours ne doit pas pouvoir activer ou désactiver le tracking xAPI.

## Modèle de données proposé

### Configuration par cours

Table proposée :

```sql
CREATE TABLE evnt_evhk_itxeb_crs_cfg (
    course_ref_id INT NOT NULL,
    course_obj_id INT NOT NULL,
    enabled TINYINT(1) NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    updated_by INT NULL,
    PRIMARY KEY (course_ref_id)
);
```

Rôle :

- `enabled = 1` : les traces xAPI peuvent être générées pour ce cours ;
- `enabled = 0` : aucune trace xAPI ne doit être générée pour ce cours ;
- absence de ligne : comportement à décider explicitement dans la stratégie par défaut.

### Configuration par ressource

Table proposée :

```sql
CREATE TABLE evnt_evhk_itxeb_res_cfg (
    course_ref_id INT NOT NULL,
    ref_id INT NOT NULL,
    obj_id INT NOT NULL,
    obj_type VARCHAR(32) NOT NULL,
    enabled TINYINT(1) NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    updated_by INT NULL,
    PRIMARY KEY (course_ref_id, ref_id),
    KEY idx_itxeb_res_cfg_obj (obj_id),
    KEY idx_itxeb_res_cfg_type (obj_type)
);
```

Rôle :

- `enabled = 1` : la ressource est traçable ;
- `enabled = 0` : la ressource ne doit pas générer de statement ;
- absence de ligne : comportement dépendant de la stratégie par défaut du cours.

## Stratégie par défaut

La stratégie recommandée pour V0.7 est prudente :

```text
Un cours non configuré ne génère pas de traces xAPI pilotées par cours.
```

Cela évite d'activer la collecte sans décision explicite du responsable de cours.

Option possible pour migration depuis V0.6 :

- prévoir un mode global temporaire `legacy_global_tracking` ;
- documenter clairement que le mode cible V0.7 est le contrôle par cours.

Décision à confirmer avant implémentation.

## Règle de filtrage avant outbox

Avant toute insertion dans `evnt_evhk_itxeb_out`, le plugin doit vérifier :

```text
1. Identifier le cours parent de l'objet.
2. Vérifier si le cours est configuré et activé.
3. Vérifier si la ressource est activée.
4. Si cours activé et ressource activée : générer le statement.
5. Sinon : ne pas insérer de ligne outbox.
```

Le filtrage doit s'appliquer aux familles suivantes :

- consultations issues de `read_event` ;
- téléchargements de fichier ;
- statuts de test ;
- autres événements de dépôt si couverts ensuite.

## Ressources affichées dans le cours

L'écran V0.7 doit lister les ressources traçables du cours :

- fichier ;
- test ;
- blog ;
- wiki ;
- lien web ;
- mediacast ;
- module HTML ;
- forum ;
- SCORM ou learning module si détecté.

Pour chaque ressource :

```text
[ ] Activer xAPI
Type ILIAS
Titre
ref_id
obj_id
Dernière décision enregistrée
```

Actions utiles :

- tout activer ;
- tout désactiver ;
- enregistrer ;
- revenir aux valeurs par défaut du cours.

## Extensions xAPI possibles en V0.7

Pour les statements autorisés, ajouter éventuellement :

```text
course_tracking_enabled = true
resource_tracking_enabled = true
tracking_decision_source = course_resource_config
```

Ces extensions ne sont pas indispensables au filtrage mais utiles pour l'audit TRAX.

## Points techniques à étudier

- Comment injecter un écran de configuration dans le contexte d'un objet cours avec un plugin EventHook ILIAS 10.
- Si un plugin EventHook seul ne suffit pas, identifier l'extension ILIAS nécessaire ou l'approche compatible.
- Comment récupérer proprement la liste des objets enfants d'un cours.
- Comment vérifier les permissions d'administration du cours.
- Comment appliquer le filtrage aux deux sources actuelles : EventHook et cron `read_event`.

## Lots de développement V0.7

### Lot 1 — Socle documentaire et branche

- Créer `v0.7` depuis `main`.
- Passer `plugin.php` en `0.7.0`.
- Ajouter ce plan de développement.
- Ajouter l'entrée changelog V0.7.

### Lot 2 — Tables de configuration

- Ajouter les tables cours et ressources dans `dbupdate.php`.
- Prévoir une méthode repository pour lire/écrire la configuration.
- Vérifier installation/migration sur serveur.

### Lot 3 — Résolution cours/ressources

- Lister les ressources d'un cours.
- Résoudre `course_ref_id`, `course_obj_id`, `ref_id`, `obj_id`, `obj_type`, titre.
- Préparer les données affichables par l'UI.

### Lot 4 — Interface dans le cours

- Ajouter l'écran ou onglet TRAX / xAPI dans le contexte cours.
- Afficher activation cours.
- Afficher activation par ressource.
- Enregistrer les choix.
- Respecter les permissions.

### Lot 5 — Filtrage de génération

- Appliquer la décision avant insertion outbox.
- Couvrir `read_event`.
- Couvrir fichier/test EventHook.
- Documenter les cas sans cours parent.

### Lot 6 — Validation serveur

- Tester cours désactivé.
- Tester cours activé avec ressource désactivée.
- Tester cours activé avec ressource activée.
- Tester fichier, test, blog, wiki, lien web, mediacast, module HTML.
- Vérifier qu'aucune ligne outbox n'est créée quand la ressource est désactivée.

## Critères de réussite V0.7

La V0.7 sera considérée prête lorsque :

- un responsable de cours peut activer ou désactiver xAPI pour son cours ;
- il peut choisir les ressources tracées ;
- une ressource désactivée ne produit aucune ligne outbox ;
- une ressource activée produit les statements V0.6 enrichis ;
- les droits d'accès empêchent un utilisateur non autorisé de modifier la configuration ;
- la documentation de validation V0.7 est complète.
