# Stabilisation — IliasTraxEventBridge v0.6

Ce document sert de checklist avant création du tag `v0.6.0` et avant éventuelle promotion de `main`.

## 1. État Git attendu

Sur Windows Git Bash et sur le serveur ILIAS :

```bash
git fetch origin --prune --tags
git switch v0.6
git pull --ff-only origin v0.6

git status --short
git branch --show-current
git log --oneline --decorate -10
grep -n '\$version' plugin.php
```

Attendu :

```text
working tree clean
v0.6
$version = "0.6.0";
```

`main` et `v0.5` doivent rester sur la stable V0.5.5 tant que le tag V0.6 n'est pas créé et validé.

## 2. Build serveur ILIAS

```bash
cd /var/www/ilias/public/Customizing/global/plugins/Services/EventHandling/EventHook/IliasTraxEventBridge
find . -name "*.php" -print0 | xargs -0 -n1 php -l

cd /var/www/ilias
sudo -u apache composer du
sudo -u apache php cli/setup.php build --yes
```

Attendu :

- aucune erreur de syntaxe PHP ;
- build ILIAS terminé ;
- plugin accessible dans l'administration.

## 3. Validation fonctionnelle minimale

Créer ou consulter dans un cours de test :

- un fichier ;
- un test ;
- un blog ;
- un wiki ;
- un lien web ;
- un mediacast ;
- un module HTML.

Lancer ensuite le cron ILIAS ou attendre son passage.

## 4. Contrôle outbox

```sql
SELECT id, event_type, obj_type, verb_id, status, retry_count, sent_at, last_error
FROM evnt_evhk_itxeb_out
ORDER BY id DESC
LIMIT 20;
```

Attendu :

- `status = sent` pour les nouvelles lignes ;
- `retry_count = 0` si aucun incident TRAX ;
- `last_error` vide.

## 5. Contrôle JSON V0.6

```sql
SELECT id, event_type, obj_type, status, statement_json
FROM evnt_evhk_itxeb_out
ORDER BY id DESC
LIMIT 1\G
```

Le JSON doit contenir selon le type de trace :

```text
statement_family
interaction_type
repository_object_family
object_title
object_url
course_title
course_url
contextActivities.parent
read_count
spent_seconds
read_event_first_access
read_event_last_access
result.duration
outbox_id
outbox_table
event_log_id
statement_uuid
event_record_source
source_table
deduplication_key
```

## 6. Contrôle wording bilingue

Vérifier que les descriptions xAPI contiennent bien deux textes distincts :

```text
fr-FR = texte français
en-US = texte anglais
```

Exemple attendu pour un module HTML :

```text
fr-FR = Consultation d’un module HTML ILIAS dans un cours
en-US = Consultation of an ILIAS HTML learning module in a course
```

Exemple attendu pour le cours parent :

```text
fr-FR = Cours parent ILIAS
en-US = Parent ILIAS course
```

## 7. Contrôle diagnostics outbox

Pour une trace issue de `read_event` :

```text
event_record_source = read_event_tracker
source_table = read_event
deduplication_key = read_event:{obj_id}:{user_id}:{last_access}:{read_count}
```

Pour une trace issue d'un EventHook :

```text
event_record_source = event_hook_log
source_table = evnt_evhk_itxeb_log
deduplication_key = event_log:{event_log_id}
```

## 8. Contrôle admin V0.6

Dans ILIAS :

```text
Administration > Plugins > IliasTraxEventBridge > Configurer
```

Vérifier la présence de :

- titre `IliasTraxEventBridge — V0.6` ;
- section `Supervision V0.6` ;
- bloc `Exploitation / maintenance` ;
- compteurs total / 24h / 7j ;
- compteurs `sent`, `generated`, `failed` ;
- familles xAPI ;
- sources techniques ;
- dernières clés de diagnostic ;
- dernières erreurs.

## 9. Contrôle anti-parasites root/crs

```sql
SELECT id, event_log_id, event_type, obj_type, ref_id, obj_id,
       user_id, status, created_at, sent_at
FROM evnt_evhk_itxeb_out
WHERE obj_type IN ('root', 'crs')
ORDER BY id DESC
LIMIT 20;
```

Attendu : aucune nouvelle ligne parasite `root` ou `crs` après les tests V0.6.

## 10. Documentation avant tag

Vérifier que les fichiers suivants sont alignés :

```text
README.md
README_TECHNIQUE.md
CHANGELOG.md
docs/VALIDATION.md
docs/OPERATIONS.md
docs/V0.6_DEV_PLAN.md
docs/V0.6_STABILISATION.md
```

## 11. Préparation du tag

À faire uniquement après validation complète :

```bash
git fetch origin --prune --tags
git switch v0.6
git pull --ff-only origin v0.6

git tag -a v0.6.0 -m "IliasTraxEventBridge v0.6.0 stable"
git push origin v0.6.0
```

Ne pas promouvoir `main` tant que le tag n'a pas été contrôlé côté serveur.

## 12. Promotion main éventuelle

Après tag validé :

```bash
git switch main
git pull --ff-only origin main
git merge --ff-only v0.6
git push origin main
```

Puis vérifier :

```bash
git log --oneline --decorate -5
grep -n '\$version' plugin.php
```

La promotion de `main` doit être une décision explicite après validation du tag.
