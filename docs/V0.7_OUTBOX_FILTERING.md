# V0.7 — Filtrage xAPI avant outbox

## Objectif

Ce jalon applique réellement la configuration V0.7 avant toute insertion dans l'outbox locale `evnt_evhk_itxeb_out`.

Le comportement devient strictement opt-in :

- le cours doit être explicitement activé dans `evnt_evhk_itxeb_ccfg` ;
- la ressource doit être explicitement activée dans `evnt_evhk_itxeb_rcfg` ;
- si l'une des deux conditions manque, aucun statement xAPI n'est généré.

## Tables utilisées

### Configuration cours

```sql
SELECT *
FROM evnt_evhk_itxeb_ccfg
WHERE course_ref_id = 194;
```

Le cours est autorisé seulement si :

```text
course_ref_id = 194
enabled = 1
```

### Configuration ressource

```sql
SELECT course_ref_id, ref_id, obj_id, obj_type, enabled
FROM evnt_evhk_itxeb_rcfg
WHERE course_ref_id = 194
ORDER BY ref_id;
```

La ressource est autorisée seulement si :

```text
course_ref_id = 194
ref_id = <ressource>
enabled = 1
```

## Chemins filtrés

### EventHook

Le filtrage est appliqué dans :

```text
classes/class.ilIliasTraxEventBridgeEventRouter.php
```

Le router continue d'enregistrer l'événement brut dans `evnt_evhk_itxeb_log` si le debug est actif, mais il refuse l'insertion outbox si le cours ou la ressource n'est pas activé.

### read_event

Le filtrage est appliqué dans :

```text
classes/class.ilIliasTraxEventBridgeReadEventTracker.php
```

Les consultations `read_event` refusées sont marquées traitées dans :

```text
evnt_evhk_itxeb_read
```

Cela évite que le cron réessaie sans fin la même consultation non autorisée.

Une consultation ultérieure pourra être retraitée si `last_access` ou `read_count` évolue après activation de la ressource.

## Règle de décision

```text
course enabled = evnt_evhk_itxeb_ccfg.enabled = 1
resource enabled = evnt_evhk_itxeb_rcfg.enabled = 1

allowed = course enabled AND resource enabled
```

En V0.7, l'absence de configuration vaut refus.

## Exemple avec le cours de test

Configuration validée :

```text
course_ref_id = 194
course_obj_id = 629
course enabled = 1
```

Ressources activées :

```text
ref_id 205 / obj_id 651 / blog / enabled = 1
ref_id 207 / obj_id 655 / htlm / enabled = 1
```

Résultat attendu après ce jalon :

```text
blog 205 -> statement xAPI autorisé
htlm 207 -> statement xAPI autorisé
autres ressources du cours 194 -> aucune insertion outbox
```

## Requêtes de contrôle avant test

```sql
SELECT *
FROM evnt_evhk_itxeb_ccfg
WHERE course_ref_id = 194;

SELECT course_ref_id, ref_id, obj_id, obj_type, enabled, updated_at, updated_by
FROM evnt_evhk_itxeb_rcfg
WHERE course_ref_id = 194
ORDER BY ref_id;
```

## Requête de contrôle outbox

Avant de tester une ressource, noter le dernier identifiant outbox :

```sql
SELECT MAX(id) AS max_outbox_id
FROM evnt_evhk_itxeb_out;
```

Après consultation d'une ressource autorisée ou refusée :

```sql
SELECT id, event_log_id, event_type, obj_type, ref_id, obj_id,
       user_id, status, created_at, sent_at
FROM evnt_evhk_itxeb_out
WHERE id > <MAX_ID_AVANT_TEST>
ORDER BY id ASC;
```

## Test attendu — ressource autorisée

Consulter une ressource activée, par exemple `ref_id = 205` ou `ref_id = 207`.

Attendu : une nouvelle ligne peut apparaître dans `evnt_evhk_itxeb_out` si l'événement est supporté ou si le cron `read_event` traite la consultation.

## Test attendu — ressource désactivée

Consulter une ressource désactivée, par exemple `ref_id = 199`, `200`, `204`, etc.

Attendu : aucune nouvelle ligne outbox ne doit être insérée pour cette ressource.

## Vérification read_event

Pour les consultations issues du cron :

```sql
SELECT obj_id, usr_id, last_access, read_count, processed_at
FROM evnt_evhk_itxeb_read
ORDER BY processed_at DESC
LIMIT 20;
```

Une ressource refusée peut apparaître dans `evnt_evhk_itxeb_read` comme traitée, sans ligne correspondante dans l'outbox.

## Limites du jalon

Ce jalon ne modifie pas encore :

- l'emplacement natif dans les paramètres du cours ;
- la structure du statement xAPI ;
- les tables outbox ;
- l'écran de supervision admin.

Il introduit uniquement la règle de décision avant insertion outbox.
