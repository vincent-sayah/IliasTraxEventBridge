# V0.9 — Détection des échecs fréquents

## Objectif

Ce lot ajoute un signal pédagogique dans l’onglet `Analyse` afin d’identifier les tests qui sont suffisamment utilisés et qui présentent un taux d’échec élevé.

L’objectif est de faire ressortir rapidement une ressource qui mérite une vérification pédagogique : consigne ambiguë, difficulté trop élevée, prérequis manquant ou problème de contenu.

## Vue concernée

La modification concerne l’onglet :

```text
Analyse
```

Elle agit sur la colonne :

```text
Signal
```

## Règle MVP

Le signal devient :

```text
échecs fréquents
```

quand les conditions suivantes sont réunies :

```text
obj_type = tst
test_attempts >= 3
taux échec >= 50 %
```

Un signal plus faible peut être affiché :

```text
à surveiller
```

quand :

```text
obj_type = tst
test_attempts >= 3
taux échec >= 30 %
aucun autre signal prioritaire n’est déjà présent
```

## Calcul

```text
taux échec = test_failed / test_attempts * 100
```

Le calcul protège contre la division par zéro.

## Pourquoi un seuil de 3 tentatives ?

Avec une seule tentative, un échec peut être accidentel.

Avec deux tentatives, le signal peut encore être trop instable.

À partir de trois tentatives, le MVP considère qu’il existe un minimum de volume pour afficher un signal pédagogique.

Ce seuil pourra être rendu configurable plus tard.

## Interaction avec les filtres

Le signal respecte les filtres existants :

- période ;
- ressource ;
- type d’objet.

Exemples :

```text
Type = Test (tst)
```

La vue Analyse affiche uniquement les tests et leurs signaux.

```text
Ressource = test ref_id 216
```

La vue Analyse affiche uniquement cette ressource, et le filtre type est ignoré.

## Implémentation

La modification est injectée pendant l’installation du companion UIHook par le script :

```text
scripts/patch_course_ui_failure_signals.php
```

Le script d’installation appelle les patchers dans cet ordre :

```text
1. patch_course_ui_type_filter.php
2. patch_course_ui_success_rates.php
3. patch_course_ui_failure_signals.php
```

Le troisième patch dépend du deuxième, car il réutilise les variables calculées pour les tentatives, réussites et échecs.

## Limites MVP

Cette détection reste volontairement simple.

Elle ne remplace pas une analyse complète du LRS TRAX. Elle exploite uniquement les traces locales générées par le plugin et disponibles dans l’outbox.
