# V0.10 — Lecture directe TRAX / LRS

## Objectif

Ajouter une lecture directe du LRS TRAX depuis l'écran `Suivi xAPI` du cours, sans remplacer l'analyse locale V0.9.1.

La V0.9.1 exploite l'outbox locale `evnt_evhk_itxeb_out`. La V0.10 ajoute un contrôle direct côté LRS afin de vérifier que les statements envoyés existent bien dans TRAX.

## Principe retenu

La lecture directe est volontairement séparée de l'envoi :

- l'envoi continue d'utiliser l'outbox locale et le client d'envoi existant ;
- la lecture utilise un nouveau client dédié, read-only ;
- aucune ligne SQL locale n'est modifiée pendant cette lecture ;
- l'écran ne doit pas être bloqué si TRAX est indisponible.

## Client ajouté

```text
classes/class.ilIliasTraxEventBridgeLrsReadClient.php
```

Ce client effectue uniquement des appels :

```text
GET /statements
```

avec les en-têtes xAPI existants :

```text
Accept: application/json
X-Experience-API-Version: 1.0.3
```

et l'authentification Basic HTTP déjà configurée pour TRAX.

Le client sait également suivre une URL `more` renvoyée par le LRS. Si l'URL `more` est relative, elle est reconstruite avec le schéma, l'hôte et le port de l'endpoint `/statements` configuré.

## Synthèse cours ajoutée

```text
classes/class.ilIliasTraxEventBridgeLrsCourseSummary.php
```

La synthèse interroge le LRS avec l'activité xAPI du cours :

```text
<ILIAS_BASE_URL>/xapi/activity/course/ref/<course_ref_id>
```

Paramètres utilisés :

```text
activity=<activité cours>
related_activities=true
since=<date début période>
limit=100
```

Le paramètre `related_activities=true` est nécessaire car les statements générés par le plugin utilisent le cours comme activité parente dans `contextActivities.parent`.

## Lot 1 — contrôle LRS minimal

Le premier bloc affiche :

- état disponible / indisponible ;
- code HTTP ;
- nombre de statements retournés ;
- activité cours xAPI utilisée ;
- erreur éventuelle.

## Lot 2 — comparaison local / LRS

Le bloc `TRAX / LRS direct` affiche maintenant aussi :

- traces locales générées ;
- traces locales envoyées ;
- statements retournés par TRAX ;
- écart entre traces locales envoyées et statements retournés par TRAX ;
- statut de comparaison ;
- indicateur `more` retourné par le LRS ;
- table des verbes retournés par TRAX.

Règle d'interprétation :

```text
écart = traces locales envoyées - statements retournés par TRAX
```

- `0` : cohérent sur la fenêtre et dans la limite de lecture ;
- positif : des traces locales envoyées ne sont pas retrouvées dans les statements LRS retournés ;
- négatif : TRAX retourne plus de statements que les traces locales envoyées de la période.

## Lot 3 — pagination LRS

La synthèse suit automatiquement le champ `more` retourné par TRAX.

Pour éviter une boucle infinie ou une lecture trop lourde, la lecture est limitée à :

```text
5 pages maximum
100 statements par page
```

Le bloc affiche désormais :

- nombre de pages LRS lues ;
- statut de pagination : `complète`, `incomplète` ou `tronquée limite sécurité` ;
- éventuelle erreur de pagination ;
- éventuel `more` restant si la limite de sécurité est atteinte.

La comparaison local / LRS devient donc plus fiable qu'au lot 2, car elle additionne les statements des pages lues au lieu de se limiter à la première réponse.

## Lot 4 — détail par ressource côté LRS

La synthèse agrège désormais les statements retournés par TRAX par ressource.

La ressource est identifiée en priorité par l'extension xAPI :

```text
/xapi/extensions/ref_id
```

Si cette extension n'est pas disponible, la clé de regroupement retombe sur `statement.object.id`.

Le tableau de bord affiche une nouvelle table :

```text
Ressources retournées par TRAX
```

Colonnes affichées :

- titre de la ressource ;
- object id xAPI ;
- type ILIAS si disponible ;
- ref_id si disponible ;
- nombre de statements retournés par TRAX.

## Lot 5 — comparaison local / LRS par ressource

La table ressource devient une vraie comparaison local / TRAX :

```text
Comparaison local / TRAX par ressource
```

Colonnes affichées :

- ressource ;
- type ;
- ref_id ;
- traces locales générées ;
- traces locales envoyées ;
- statements retournés par TRAX ;
- écart ;
- statut.

Règle d'interprétation :

```text
écart ressource = traces locales envoyées - statements TRAX de la ressource
```

Statuts possibles :

- `cohérent` ;
- `absent TRAX` ;
- `local > TRAX` ;
- `TRAX > local` ;
- `TRAX sans ref_id`.

## Lot 6 — divergence par statement UUID

La synthèse LRS conserve désormais les UUID des statements retournés par TRAX.

Le tableau de bord ajoute une section :

```text
Divergence par statement UUID
```

Elle compare :

- les UUID locaux avec statut `sent` ;
- les UUID réellement retournés par TRAX dans les pages LRS lues.

Indicateurs affichés :

- UUID locaux envoyés ;
- UUID TRAX ;
- absents TRAX ;
- en plus TRAX.

Si des écarts existent, une table affiche les premiers UUID concernés avec :

- type d'écart ;
- UUID ;
- ressource ;
- verbe.

Limite actuelle : les UUID locaux utilisés pour cette comparaison proviennent des lignes expert locales chargées dans le dashboard. Pour les cours très volumineux, ce contrôle est donc indicatif tant que la collecte UUID locale exhaustive n'est pas ajoutée.

## Installation

La génération du companion applique désormais :

```text
scripts/patch_course_ui_lrs_direct_summary.php
```

via :

```bash
bash scripts/install_course_ui_companion.sh
```

ou via le wrapper Delos :

```bash
bash scripts/install_course_ui_companion_with_standalone_fix.sh
```

Le wrapper Delos applique également un fixer post-install dédié en dernier :

```text
scripts/fix_course_ui_lrs_direct_after_install.sh
```

## Limites actuelles

Ce lot ne remplace pas encore l'analyse locale. Il ajoute un contrôle LRS direct, une comparaison synthétique, une pagination contrôlée, une comparaison par ressource et une première comparaison par UUID.

Les enrichissements prévus pour les lots suivants sont :

- collecte locale exhaustive des UUID hors limite `expert_rows` ;
- export expert enrichi local + LRS.
